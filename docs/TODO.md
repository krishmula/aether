## Phase 1A: Foundation Fix

### 1. Fix NodeAddress to be hostname-based

**File:** `aether/network/node.py`
**What:** Remove `socket.gethostbyname()` from `_normalize_host()`. Store the host string as-is. Identity is `(host_string, port)` not `(resolved_ip, port)`.

**Why this matters:** Docker assigns a new IP to each container on every run. `NodeAddress("broker-2", 8000)` today resolves to `172.18.0.5`. After a broker crash and container restart, `broker-2` gets `172.18.0.6`. Any code holding the old `NodeAddress` (peer lists, snapshots, subscriber mappings) has a stale IP and can no longer route correctly. Hostname-based identity is stable across container restarts — Docker DNS always resolves `broker-2` to whoever is currently running with that hostname. This is the same identity model used by Kubernetes pods (addressed by service name, not pod IP).

**What changes:** `_normalize_host()` becomes `return host` (one line). `__eq__` and `__hash__` already use `self.host` and `self.port`, so they're correct once host is the raw string.

**Risk:** The existing integration tests use `NodeAddress("127.0.0.1", port)` — these still work because `"127.0.0.1"` is already an IP string. No regression. One test file to add: `tests/unit/test_node_address.py`.

**Do this first.** Everything else in Phase 1 touches `NodeAddress`. Getting the identity model right before any recovery code is written prevents a class of subtle bugs.

## Phase 1B: Core Failover Infrastructure

### 2. HealthMonitor asyncio task

**File:** `aether/orchestrator/health.py` (NEW)
**What:** An asyncio task that runs inside the FastAPI event loop. Polls each broker's `/status` endpoint every 5 seconds using `httpx.AsyncClient`. Tracks consecutive failure counts per broker. After 3 consecutive failures, calls `on_broker_dead(broker_id, host, port)`.

**Why asyncio (not threading):** The orchestrator is a FastAPI app — it runs entirely in an asyncio event loop. Adding a background `threading.Thread` that reads and writes `_components` creates a data race (the dict has no lock). Keeping everything in the event loop avoids this entirely. `asyncio.gather` also lets us poll all N brokers concurrently for free — with sequential threading, 10 brokers × 2s timeout = 20s worst-case latency before detecting a failure.

**Why httpx (not urllib.request):** `urllib.request` blocks the event loop. `httpx.AsyncClient` is the standard async HTTP client for FastAPI applications. It's already in the Python ecosystem, well-tested, and a one-line `await client.get(url, timeout=2.0)` call.

**Key design:** Consecutive failure count resets on any successful response. "Consecutive" means network error OR non-200 response. This handles the case where a broker is up but returning errors — it should trigger failover too.

**Wired in:** `main.py` startup event adds the HealthMonitor as `asyncio.create_task()`. On_broker_dead callback invokes `RecoveryManager.handle_broker_death()`.

**Test:** `tests/unit/test_health_monitor.py` — mock `httpx.AsyncClient`, verify callback fires after 3 failures, verify counter resets on success.

### 3. POST /recover on broker status server

**File:** `aether/gossip/status.py`
**What:** Add `do_POST()` to `_StatusHandler`. Handle `POST /recover` with JSON body `{"dead_broker_host": "broker-2", "dead_broker_port": 8000}`. Calls `broker.request_snapshot_from_peers(dead_broker_addr)` then `broker.recover_from_snapshot(snapshot)`.

**Why not a new HTTP framework:** The broker's status server is intentionally minimal — stdlib `ThreadingHTTPServer`, no dependencies. The plan (from /office-hours) specified "do not add a new port." Adding `do_POST` to the existing handler is 30 lines and keeps the broker's HTTP surface minimal. The data plane (gossip TCP) and control plane (HTTP) stay separated.

**Error cases to handle:**
- Missing/invalid JSON body → 400
- `request_snapshot_from_peers` returns None (no peers have the snapshot) → 500 with body `{"error": "no_snapshot_available"}`
- Any exception during recovery → 500

**Why the broker handles recovery, not the orchestrator calling broker Python directly:** The broker runs in a Docker container — it's a separate process. The only interface is HTTP. The orchestrator can't call broker methods directly.

**Test:** `tests/unit/test_broker_status_extended.py` — mock `GossipBroker`, verify 200/400/500 paths.

### 4. GET /snapshots/{host}/{port} on broker status server

**File:** `aether/gossip/status.py`
**What:** Add path routing to `_StatusHandler.do_GET()`. Handle `GET /snapshots/{host}/{port}` by looking up `broker._peer_snapshots[NodeAddress(host, int(port))]` and returning the serialized snapshot as JSON.

**Why add to broker's status server (not orchestrator):** The orchestrator doesn't have direct access to `_peer_snapshots` — that lives inside each broker process. The orchestrator needs to query surviving brokers for what snapshots they've stored. The existing `host_status_port` mapping (18010, 18020, etc.) is already exposed to the Docker host network, so no new ports needed.

**Path parsing:** `parts = path.split("/")` → validate `len(parts) == 3` and `parts[0] == "snapshots"` → return 400 on malformed paths to prevent `IndexError`.

**Serialization:** `BrokerSnapshot` is a `@dataclass`. Serialize to JSON with `dataclasses.asdict()` plus custom handling for `NodeAddress` objects (serialize as `"{host}:{port}"` strings). The orchestrator deserializes and checks `snapshot.timestamp` for freshness.

**Test:** included in `test_broker_status_extended.py`.

### 5. RecoveryManager

**File:** `aether/orchestrator/recovery.py` (NEW)
**What:** `RecoveryManager.handle_broker_death(broker_id, host, port)` decides which recovery path to use and executes it.

**Decision logic:**
1. Query each surviving broker's `GET /snapshots/{host}/{port}` (async, httpx, concurrent)
2. Find the freshest snapshot: compare `snapshot.timestamp` to `time.time()`
3. If `time.time() - snapshot.timestamp < 30.0`: Path A (replacement)
4. Otherwise: Path B (redistribution)

**Path A — Replacement (fresh snapshot):**
1. Remove the dead broker container via `DockerManager.remove_broker()`
2. Create a replacement container with the same `broker_id` via `DockerManager.create_broker()`
3. Poll the replacement's `GET /status` until it returns 200 (with retry, see item 6B)
4. Send `POST /recover` with the dead broker's address
5. Update `_components[container_name].status = RUNNING`
6. Emit `BROKER_RECOVERED(recovery_path='replacement')` WebSocket event

**Path B — Redistribution (stale/no snapshot):**
1. Remove the dead broker container
2. Find orphaned subscribers from `_components.values()` where `c.broker_id == dead_broker_id`
3. Assign each to the surviving broker with the fewest current subscribers (load balancing)
4. Update `_components[sub].broker_id` for each reassigned subscriber
5. Emit `BROKER_RECOVERED(recovery_path='redistribution')` WebSocket event

**Why 30s threshold:** The `snapshot_interval` in `config.docker.yaml` is 30s. The failure detection window is 15s (3 × 5s polls). So at the moment of death detection, the last snapshot is 0-30s old. The 30s threshold means Path A is available roughly 50% of the time in steady state — enough to demonstrate it in the demo. (See also item 6B — retry waits for broker readiness, so Path A is reliable when a fresh snapshot exists.)

**Emit on detection:** Before choosing a path, emit `BROKER_DECLARED_DEAD` event. This is what the dashboard + Grafana pick up to show the failure event.

**Test:** `tests/unit/test_recovery_manager.py` — mock `DockerManager` + `httpx.AsyncClient`, verify path A selected when snapshot age < 30s, path B when stale/missing.

### 5B. RecoveryManager retry-with-backoff for POST /recover

**File:** `aether/orchestrator/recovery.py`
**What:** After creating the replacement container (Path A), don't immediately send `POST /recover`. Instead: poll `GET /status` on the new broker every 500ms until it returns 200 or timeout (10s). Only then send `POST /recover`.

**Why this is required (not optional):** Docker starts the container, but the broker process inside it takes 2-5 seconds to start its HTTP server. The first `POST /recover` will get `ConnectionRefused`. Without this wait: Path A silently fails (orchestrator logs an error, no recovery happens, subscribers stay disconnected). This isn't a future enhancement — it's a correctness requirement.

**Pattern:** Re-use the same `httpx.AsyncClient` from HealthMonitor. `async for attempt in range(20): try GET /status; if 200: break; await asyncio.sleep(0.5)`. Raise after timeout.

**Test:** `test_recovery_manager.py` — mock httpx to return 503 twice then 200; verify `POST /recover` is sent only after the 200.

### 6. Bootstrap DELETE /deregister endpoint

**File:** `aether/gossip/bootstrap.py`
**What:** Add `DELETE /deregister` to the bootstrap server's HTTP handler. Body: `{"host": "broker-2", "port": 8000}`. Removes the matching address from `registered_brokers`.

**Why:** Bootstrap currently only adds brokers — never removes them. After a broker dies, its address stays in `registered_brokers` forever. When the replacement broker starts and queries bootstrap for peers, it gets the dead broker's address back. The replacement then tries to gossip to a dead IP, logs errors, and the peer list looks wrong in the dashboard.

**Recovery manager calls this:** After removing the dead container, before spinning up the replacement, call `DELETE /deregister` on bootstrap. This ensures the replacement gets a clean peer list on startup.

**Test:** `test_broker_status_extended.py` or new `test_bootstrap_deregister.py`.

### 7. GET /api/assignment endpoint on orchestrator

**File:** `aether/orchestrator/main.py`
**What:** `GET /api/assignment?subscriber_id={id}` — looks up the subscriber in `_components`, returns `{"broker_host": "broker-1", "broker_port": 8000}`.

**Why:** This is the "pull-based recovery" endpoint — the same pattern Kafka consumer groups use. Subscribers query this when they detect their broker is dead. The orchestrator is the single source of truth for which subscriber is assigned to which broker. This is why we're using pull over push: if a push notification (`BrokerRecoveryNotification`) is missed (network error, subscriber busy), the subscriber can still recover by polling this endpoint.

**Response fields:** `broker_host` should be the Docker hostname (e.g., `"broker-1"`) not the IP. Subscribers create `NodeAddress(broker_host, broker_port)` — after the NodeAddress fix (item 1), this resolves correctly to the current container.

**Error cases:** `subscriber_id` not found → 404. Subscriber has no `broker_id` yet → 404. Don't return a stale broker that's been declared dead — check `_components[sub].broker_id` refers to a broker with `status != STOPPED/ERROR`.

**Test:** `tests/unit/test_assignment_endpoint.py` — FastAPI TestClient.

### 8. Pass AETHER_ORCHESTRATOR_URL to subscriber containers

**File:** `aether/orchestrator/docker_manager.py`, `aether/orchestrator/settings.py`
**What:** Add `orchestrator_url: str = "http://orchestrator:8001"` to `OrchestratorSettings`. Pass `AETHER_ORCHESTRATOR_URL=http://orchestrator:8001` as an environment variable in `create_subscriber()`.

**Why:** Subscribers run inside Docker containers and need to reach the orchestrator to query `/api/assignment`. The orchestrator is also a container on `aether-net`. Without this env var, subscribers have no way to know the orchestrator's address at runtime.

**Why `orchestrator` hostname:** The orchestrator Docker container should have `hostname: orchestrator` in `docker-compose.yml`. All containers on `aether-net` can reach it at `http://orchestrator:8001`.

**Settings pattern:** Matches how `settings.py` already handles `bootstrap_host` — one setting, injected via env var in the container.

### 9. Add 4 new EventType values to models.py

**File:** `aether/orchestrator/models.py`
**What:** Add to `EventType` enum:
```python
BROKER_DECLARED_DEAD = "broker_declared_dead"
BROKER_RECOVERY_STARTED = "broker_recovery_started"
BROKER_RECOVERED = "broker_recovered"        # data: {recovery_path: 'replacement'|'redistribution'}
SUBSCRIBER_RECONNECTED = "subscriber_reconnected"
```

**Why:** The WebSocket broadcaster streams events to the React dashboard and to Grafana (via log queries). Without these event types, killing a broker shows nothing in the dashboard except "broker removed" (the generic removal event). The failover story — detect → decide → execute → recover — is invisible. The Grafana Loki queries in Phase 2 filter on `broker_declared_dead` and `subscriber_reconnected` to build the failover timeline panel.

**No code change to EventBroadcaster itself** — it already sends arbitrary `dict` payloads. Just add the event types to the enum and emit them in RecoveryManager.

## Phase 1C: Client Resilience

### 10. Subscriber broker health monitoring + reconnect

**File:** `aether/network/subscriber.py`, `aether/snapshot.py`
**What:** Two additions:

**a) Ping/Pong health check**
Add `Ping` and `Pong` dataclasses to `snapshot.py`. Add `_broker_health_loop` background thread to `NetworkSubscriber` that sends `Ping` to `self.broker` every 5s and tracks when the last `Pong` was received. After 15s without a pong (3 missed), calls `_start_reconnect(self._connection_epoch)`.

**b) Generation counter reconnect**
Add `_connection_epoch: int = 0` and `_epoch_lock: threading.Lock` to `NetworkSubscriber`. `_start_reconnect(epoch)` checks under the lock whether `epoch == self._connection_epoch`. If yes, increment epoch and own the reconnect. If no, return silently (stale reconnect attempt).

The reconnect sequence (stop-reconnect-resume):
1. Stop the receive loop (`self.running = False`, join thread)
2. Connect to new broker (from `/api/assignment`)
3. Send `SubscribeRequest` for each `self.subscriptions` entry
4. Restart the receive loop

**Why stop-reconnect-resume (not concurrent subscribe):** `subscribe()` and the receive loop share the same socket receive queue. `subscribe()` sends `SubscribeRequest` and waits for `SubscribeAck` via `self.node.receive()`. If the receive loop is running, it consumes the `SubscribeAck` first. The guard at `subscriber.py:62` enforces this. Stopping the loop briefly is the correct, explicit, testable solution. The "gap" is acceptable — there's already a message gap during broker failure.

**Why generation counter (not boolean flag):** Kafka's consumer group rebalancing, LinkedIn Brooklin, and most production reconnect systems use an epoch/generation to serialize concurrent reconnect attempts. A boolean flag has undefined behavior when two threads both check "am I reconnecting?" simultaneously. The epoch makes one thread win cleanly — whoever increments the epoch owns the reconnect. Stale threads self-discard.

**Update `__slots__`:** Add `_broker_health_loop`, `_last_pong`, `_connection_epoch`, `_epoch_lock`, `_orchestrator_url` to the existing `__slots__` tuple. Forgetting this causes silent dict fallback.

**Also update `_handle_broker_recovery()`:** When `BrokerRecoveryNotification` arrives (fast path), call `_start_reconnect(self._connection_epoch)` instead of just updating `self.broker`. This ensures the same idempotency for the push path.

**Test:** `tests/unit/test_subscriber_reconnect.py`.

### 11. Publisher skip-dead-broker with cooldown

**File:** `aether/network/publisher.py`
**What:** Add `_dead_brokers: dict[NodeAddress, float]` (address → timestamp of failure). In the send loop, skip brokers in `_dead_brokers` where `time.time() - timestamp < 30.0`. On connection error, add to `_dead_brokers`. After cooldown, retry — if successful, remove from `_dead_brokers`.

**Why:** Publishers hold a fixed `broker_addresses` list and retry all of them on every publish. After a broker fails, publishers keep attempting connection to the dead IP. Without this, publishers log connection errors every 1-2 seconds and the demo looks broken even after subscriber recovery is complete.

**Why local cooldown (not orchestrator query):** The publisher doesn't need to know about the orchestrator. It just needs to route around failed connections. The cooldown pattern is simpler, has no network dependency, and handles all failure modes (not just broker death — also temporary network issues). This is how most production message clients work (e.g., Kafka producer `reconnect.backoff.ms`).

**30s cooldown:** Matches the snapshot freshness threshold. After 30s, the replacement broker is definitely running (if Path A was taken). If Path B was taken, the dead broker won't be replaced, but the cooldown prevents infinite error logging.

**Test:** `tests/unit/test_publisher_cooldown.py` (new unit test — straightforward mock of NetworkNode.send).

## Phase 2: Observability Stack

Phase 2 must match the repo as it exists now:

- brokers, publishers, and subscribers are dynamically created by the orchestrator
- the repo already has structured JSON logging and async log dispatch
- the repo already has `/status`, `/api/metrics`, and WebSocket event surfaces

So the observability job is not just "ship logs to Loki." It is to unify logs, metrics, and recovery events into one coherent operational story.

### 18. scripts/chaos.sh

**File:** `scripts/chaos.sh` (new), `scripts/reset.sh` (new)
**What:** `chaos.sh` kills a random running broker container, waits for recovery to complete, logs timing, then auto-resets after 60s.

**Idempotency requirements:**
1. Check if target container is already stopped before killing (skip if already dead)
2. Poll the dead broker's port until it's unreachable (confirm death)
3. Poll `/api/state` on the orchestrator until broker shows RECOVERED status
4. Log the wall-clock time from kill to recovery
5. After 60s, restart the container (or call `scripts/reset.sh`)
6. The 60s timer waits for recovery completion — it doesn't start until the orchestrator reports RECOVERED

**Why poll for completion (not sleep):** Fixed sleeps are fragile — too short and the demo resets before recovery finishes, too long and the demo waits unnecessarily. Polling the orchestrator's `/api/state` endpoint gives an exact recovery timestamp. This also makes the timing printout accurate and impressive.

## Known Limitations (honest about what this isn't)

Document these in README.md. Being upfront about limitations is a signal of engineering maturity:

- **Not production-grade:** No authentication, no TLS, no persistence. Messages are dropped on broker failure in Path B (redistribution).
- **Single orchestrator:** No HA for the orchestrator itself. If it crashes, failover can't be triggered until it restarts.
- **No message replay:** Subscribers that missed messages during failover don't get them back.
- **Docker-centric demo:** The main demo path depends on Docker container lifecycle management. It demonstrates the concepts, not a production-ready control plane.
